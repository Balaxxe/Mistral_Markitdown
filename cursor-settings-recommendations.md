## Security (apply these first)

### 1. Destructive-command approval
Settings → Agent → **Require approval for destructive commands**. Forces click-through on `rm -rf`, `DROP TABLE`, `git push --force` while auto-approving tests/builds/reads. This complements the hook-based shell guard (defense in depth — hooks can be edited, this setting cannot be edited by the agent).

### 2. Sandbox — know the default and the Windows bug
The terminal sandbox is **OFF by default on every OS**. Enable: Settings → Agent → Security → **Enable Sandbox Mode**, then set Network Access (Settings → Agent → Network Access) to `Restrict to allowlist + Cursor defaults`.

**Windows caveat (this is your dev OS):** with Auto-Run in Sandbox enabled, the command allowlist is **silently ignored** (open bug). Workaround: enable **Legacy Terminal Tool** (Settings → Agents → Inline Editing & Terminal) if you depend on allowlist behavior.

### 3. Workspace Trust + Privacy Mode
Enable Workspace Trust (`"security.workspace.trust.enabled": true`) so untrusted repos can't auto-execute tasks — this also gates project hooks. Enable Privacy Mode (Settings → Privacy) if you work on proprietary code.

### 4. Hooks observability
After any hooks change, verify in **Customize → Hooks tab** and the **Hooks output channel**. Cursor hot-reloads hooks.json; restart if hooks don't appear (known Windows quirk after edits). Shell-guard denials log to `~/.cursor/logs/shell-guard.jsonl`.

### 5. Stop-hook caveat on Windows
The scoped-test stop hook runs pytest on changed files when the agent finishes. On Windows an open Cursor bug sometimes drops the auto-continue message — tests still run and results land in the temp log the hook names, but the agent may not self-correct. CI (GitHub Actions) is the reliable backstop.

## Agents Window workflow

### 6. Open the project as a workspace — always
`.cursor/rules/` and project config only load when a workspace is open. An agent started from the bare home directory sees none of your project rules.

### 7. Parallel agents and isolation
Up to 8 parallel agents, each in its own git worktree or cloud VM. Useful commands: `/multitask` (fan a task out to subagents), `/side` or `/btw` (side chat without interrupting the main agent), `/in-cloud` (send a subtask to a cloud VM on its own branch).

### 8. Plan Mode
Use Plan Mode for multi-file refactors (e.g. splitting `mistral_converter.py`), migrations, and CI changes; direct execution for narrow edits.

## Context hygiene

- Long chats dilute rules — start a fresh agent per task; re-state critical constraints in prompts on long sessions.
- Keep `.cursorindexingignore` maintained (semantic-search noise) and let big tool outputs go to files.
- `tests/test_mistral_converter.py` is ~170KB — point agents at specific test classes rather than the whole file.

## Tab

- If Tab fights indentation, rebind "Accept Cursor Tab Suggestion" (e.g. `Ctrl+.`) — still the highest-ROI keybinding change.
- Enable partial accepts; disable suggestions in comments if noisy.
- **Windows:** never add `beforeTabFileRead`/`afterTabFileEdit` hooks — per-keystroke process spawns back up the completion queue (open perf bug).

## Network

- HTTP/2 default; drop to HTTP/1.1 only for VPN/proxy connection failures (Settings → Network).

## Cloud agents

- `.cursor/environment.json` defines the VM (poppler + ghostscript + Python deps). Keep `install` idempotent.
- Add `MISTRAL_API_KEY` in the dashboard **Secrets** tab if a cloud task needs live OCR — tests don't need it.

## User Rules (Settings → Rules)

Keep short and personal; repo policy belongs in AGENTS.md / project rules:

```
- Proceed on clear, reversible, low-risk edits; ask before destructive or external-side-effect actions.
- Be concise. End with: files changed, validation run, remaining risks.
- Do not commit or push unless I explicitly ask.
```
